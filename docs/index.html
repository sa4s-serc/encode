Abstract

Energy consumption of software has become a first-class engineering concern, yet existing profiling tools operate only at runtime, reporting coarse process- or method-level metrics. Individual constructs such as loops, conditionals, and functions — which execute in microseconds — remain invisible to standard profilers.

We present EnCoDe, a methodology for fine-grained, design-time energy estimation of Python code blocks — no execution required. Our contributions are threefold: (1) PowerLens, a novel measurement infrastructure that achieves reliable sub-millisecond energy readings by combining execution amplification, RAPL temporal synchronization, and IQR-based statistical aggregation; (2) an empirical study of over 18,000 Python programs producing a dataset of 14,000+ block-level energy measurements spanning six orders of magnitude; and (3) ML-based predictive models achieving R² = 0.755 for energy regression and 80.6% accuracy for Low / Medium / High tier classification using 33 static AST features.

The resulting IDE integration — WattWise — brings energy awareness to developers as a lint-like design-time signal, enabling proactive optimization before code ever runs.

Key Contributions

🔬

PowerLens

A novel sub-millisecond energy measurement methodology overcoming RAPL's 1 ms granularity limit through five coordinated mechanisms: environmental stabilization, execution amplification, temporal synchronization, calibrated subtraction, and IQR-based statistical aggregation. EnCoDe Dataset

8,358 Python code blocks (6,686 train / 1,672 test) extracted from 18,612 real-world programs, each with ground-truth energy measurements spanning six orders of magnitude (2.37×10⁻⁵ J – 7.48×10² J). Annotated with 33 static AST features across 7 categories.

WattWise

Methodology

Figure: The EnCoDe framework. Python source files are parsed into ASTs to extract code blocks; PowerLens measures ground-truth energy; 33 static features are extracted and used to train regression and classification models; at inference time, new code is analysed without execution.

Pipeline Steps

1

Block Identification

Python ASTs are parsed; def, for, while, if, try, and with nodes are extracted as independent blocks.

2

PowerLens Measurement

Each block is executed N×, aligned to RAPL update boundaries, calibrated for loop overhead, and aggregated across 10 IQR-filtered trials to produce a stable energy signature.

3

Feature Extraction

33 static AST metrics per block — complexity, density, Halstead, structural, and pattern counts — without executing the code.

4

ML Training

Gradient Boosting (regression, sqrt-transformed target) and XGBoost (Low/Med/High classification) trained with stratified k-fold cross-validation.

5

Design-Time Inference

New code is parsed, features extracted, and models applied inline in VS Code — energy estimated in under 1 second with no code execution.

PowerLens: Sub-Millisecond Energy Measurement

RAPL (Intel's Running Average Power Limit) updates at ~1 ms intervals, but individual code blocks execute in microseconds — causing >110% coefficient of variation and frequent zero readings. PowerLens solves this through five coordinated mechanisms:

  • ① Environmental Stabilization — lock CPU frequency, disable Turbo Boost, pin to single core
  • ② Execution Amplification — repeat block N× to push cumulative energy above RAPL threshold
  • ③ Temporal Synchronization — align execution start with RAPL counter update boundary
  • ④ Calibrated Subtraction — remove pre-measured loop overhead via linear calibration
  • ⑤ Statistical Aggregation — 10 trials with IQR outlier removal; mean of stable observations

Result: >90% of blocks achieve <10% coefficient of variation. Aggregated PowerLens measurements closely match coarse-grained PyRAPL readings at substantially lower variance.

PowerLens measurement mechanism: synchronization, amplification, and calibrated subtraction working together to extract reliable sub-millisecond energy signals.

Validation: PowerLens aggregated block measurements align with coarse-grained PyRAPL readings while showing substantially lower variance — confirming accuracy at microsecond granularity.

WattWise — VS Code Extension

EnCoDe is deployed as WattWise, a VS Code extension that brings design-time energy awareness directly into the developer's IDE. Energy estimates appear as inline decorations alongside every analysed code block — no profiler, no execution, no overhead.

WattWise annotates every def, for, while, if, try, and with block with an energy estimate and Low / Medium / High tier — powered by the EnCoDe ML models.

⚡
Inline energy decorations

Energy in Joules and tier label shown at the end of every block — right where the code lives.

🤖
AI-powered suggestions

Gemini 2.5 Flash explains why a High-energy block is expensive and proposes concrete rewrites — algorithmic improvements, vectorisation, data structure changes.

📊
Repo-wide dashboard

FastAPI + React dashboard scans an entire repository, tracks energy trends over time, and estimates annual electricity cost.

🔀
GitHub PR bot

Automatically flags energy regressions on pull requests and requests manager approval when configurable thresholds are exceeded.
